# 相談テンプレート

このテンプレートは、毎回すべての項目を埋めるためのものではありません。

必要な項目だけを短く書けばよく、未記入の項目があっても問題ありません。

迷った場合は、「今日のテーマ / 目的 / 自分の理解 / 詰まっている点 / 欲しい支援レベル」だけでも十分です。

## 1. 今日のテーマ

例：

- DB設計
- API設計
- Spring Boot実装
- テスト設計
- 認証・認可
- Docker
- GitHub Actions
- README・ADR
- 概念理解

## 2. 今やっていること

今どの作業をしているかを簡潔に書く。

例：

- 申請作成APIの責務を整理している
- 承認ルートのDB設計を考えている
- ControllerとServiceの責務分担で迷っている
- Spring Securityの設定を確認している
- テストケースの切り方を考えている

## 3. 目的

この相談で何を前に進めたいかを書く。

例：

- APIの責務を整理したい
- DB設計の違和感をレビューしてほしい
- 実装方針が大きくズレていないか確認したい
- テスト観点の抜け漏れを確認したい
- 概念理解が合っているか確認したい

## 4. 背景 / 前提

この相談に必要な前提を書く。

例：

- 今はPhase 1としてバックエンド実装を進めている
- OpenAPIは先に作成済み
- 認証方式はまだ仮置き
- 今回はUIではなくAPI責務を優先したい
- 既存のADRではこの方針になっている

## 5. 自分の理解

現時点で自分がどう理解しているかを書く。

「合っている自信はないが、こう考えている」でよい。

ここを省くと、AIが完成形を出しやすくなるため、できるだけ自分の理解を書く。

例：

- Controllerはリクエストを受け取り、Serviceに処理を渡す層だと理解している
- 認可はSecurityConfigだけで完結せず、DBを見ないと判断できないものはService側で見る必要があると考えている
- 監査ログは通常の業務データとは目的が違うので、独立して扱うべきだと考えている

## 6. 自分の案

自分なりの案を書く。

設計相談なら候補A/B、実装相談なら処理の流れ、概念相談なら自分の解釈を書く。

例：

- 案A：申請作成時に承認ルートも同時に確定する
- 案B：申請作成時はdraftだけ作り、submit時に承認ルートを確定する
- Controllerでは入力チェックとService呼び出しだけを行い、業務判断はServiceに寄せる
- 認可チェックは、粗い認証済みチェックをSecurityConfigで行い、細かい権限判定はService側で行う

## 7. 詰まっている点

どこで止まっているかを具体的に書く。

例：

- ControllerとServiceの責務の境界が分からない
- どの単位でDTOを分けるべきか迷っている
- 404と403の切り分けに迷っている
- DB制約で守るべきか、アプリ側で守るべきか判断できていない
- テストをどの粒度で書くべきか分からない

## 8. ここまでに考えたこと / 試したこと

何を考えたか、何を試したかを書く。

何も試していない場合は、「まだ試していない」でよい。

例：

- OpenAPIの定義は確認した
- 既存のADRは確認した
- Controllerだけ仮実装してPostmanで確認した
- 公式ドキュメントはまだ読めていない
- 似た処理の既存コードを見たが、今回のケースに当てはめてよいか分からない

## 9. 欲しい支援レベル

希望する支援レベルを選ぶ。

- レベル1: 問い返しだけ
- レベル2: ヒント中心
- レベル3: 骨子・観点整理まで
- レベル4: 部分例まで
- レベル5: 完全解答OK

指定がない場合、AIはレベル2として扱う。

## 10. 完成形の扱い

完成形をどこまで出してよいか指定する。

- 完成形は出さないでほしい
- まだ完成形は不要。部分例まで
- 最後に比較用の完成例を出してよい
- 今回は完成例を出してよい

## 11. レビューしてほしい観点

必要なものだけ選ぶ / 追記する。

例：

- 要件理解
- ドメイン理解
- DB正規化・制約
- API責務
- HTTP設計
- 認証・認可
- 権限制御
- 監査ログ
- 命名
- 責務分割
- 例外処理
- テスト観点
- 可読性
- 将来拡張性
- 実務的なWebアプリらしさ
- README・ADRに残すべき論点

## 12. 回答の希望形式

必要な場合だけ指定する。

指定がない場合、AIは「結論 → 主要論点 → 次の1手」の最小構成で返す。

例：

- 箇条書き中心で返してほしい
- まず結論だけ短く返してほしい
- 良い点と懸念点に分けてほしい
- 次に考えるべき問いを出してほしい
- 修正版ではなくレビューだけしてほしい
- コード例は出さず、考え方だけ教えてほしい

## 13. 添付物 / 対象

コード、SQL、API案、ER図案、README草案などを貼る。

長い場合は、「どこを見てほしいか」も書く。

例：

- 以下のControllerだけ見てほしい
- Serviceの責務分割だけ確認してほしい
- OpenAPIの該当パスだけレビューしてほしい
- ER図全体ではなく、request周りの設計だけ見てほしい
- READMEの技術選定理由だけ見てほしい

## 14. 最後にしてほしいこと

締めで欲しいものを1つだけ書く。

11や12と重複する場合は省略してよい。

例：

- 次にやるべき1手を示してほしい
- 私に確認質問を1つだけ出してほしい
- この理解で合っているか判定してほしい
- どこから直すべきか優先順位を付けてほしい
- ADRに残すべき論点だけ抽出してほしい
